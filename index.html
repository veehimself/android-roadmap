Tier 1 — Deep mastery (~35%) · you use this every single day
Null safety — ?, ?., ?:, !!. Android APIs express absence through types, not exceptions. String? vs String appears on every screen, every ViewModel, every API model.
Data classes + sealed classes + enum classes — data classes for API responses, UI state, navigation args. Sealed classes as state machines (Loading | Success | Error). Enum classes for fixed sets like user roles, sort orders, tab identifiers.
Lambdas + function types — (String) -> Unit, () -> Boolean. Compose passes UI callbacks as function parameters everywhere. You need to write and read these fluently, not just recognise them.
Coroutines — all of it — suspend, launch, async/await, withContext, Dispatchers, Flow, StateFlow, SharedFlow, channelFlow, structured concurrency, cancellation. Not just the basics — you need to understand why viewModelScope cancels when the ViewModel dies, and why you use Dispatchers.IO for Room/network calls.
Scope functions — let, run, apply, also, with. Android code uses these constantly for null checks (value?.let { ... }), builder patterns (apply { ... }), and chaining operations. Know which one returns the receiver vs the lambda result.
Collections — map, filter, flatMap, fold, groupBy, first/firstOrNull, any/all/none, sortedBy. You'll transform API responses into UI models constantly. These need to be fast to write.
Extension functions + object + companion object — fun Context.toast(msg: String), singleton instances, factory methods. You'll write and read these throughout every Android codebase.
val/var, lateinit, lazy — immutability by default, deferred initialisation, lazy computation. private val _state = MutableStateFlow(...) vs val state = _state.asStateFlow() is a pattern you write on every ViewModel.
Default + named parameters + destructuring — default parameters eliminate builder boilerplate. Named params make composable function calls readable. Destructuring unpacks data classes: val (id, name) = product.
when expressions — exhaustive branching on sealed classes is a core pattern. The compiler enforces that every state is handled — no missed else that silently swallows a new state.
Tier 2 — Working knowledge (~20%) · you'll encounter this regularly
Property delegation — by lazy, by viewModels() — by lazy for expensive one-time computation. by viewModels() is literally how you access a ViewModel in a Fragment. You'll use these without thinking about the delegation mechanism underneath.
Inline functions + reified — Compose internally uses inline functions for performance (composables are inlined at compile time). reified lets you access generic type info at runtime without reflection: inline fun <reified T> intent() = Intent(this, T::class.java). You'll write some of these in utility code.
Generics basics — type bounds, basic variance — you'll constantly work with Flow<List<Product>>, Result<T>, StateFlow<UiState>. You need to write generic repository functions like fun <T> safeCall(block: suspend () -> T): Result<T>. You don't need to write complex variance declarations, but you need to read them in library code.
Interfaces + abstract classes — Repositories, UseCases, and DataSources are defined as interfaces so they can be swapped in tests. interface ProductRepository with a ProductRepositoryImpl — this pattern is everywhere in clean architecture.
Type aliases — typealias OnClick = () -> Unit. Used to name complex function types so composable signatures stay readable.
Annotations basics — @Composable, @HiltViewModel, @Inject, @Serializable, @Entity, @Dao. You don't need to write annotation processors — you just need to use them correctly and understand what each one tells the compiler or framework to do.
Tier 3 — Skim-level (~10%) · know it exists, learn when you hit it
Operator overloading — Compose uses + to combine Modifiers (Modifier.padding(8.dp) + Modifier.fillMaxWidth()). You won't write operator overloads, but you read them. Knowing operator fun plus() exists removes the confusion.
Covariance / contravariance (in/out) — you'll see Flow<out T> in library source code. You don't need to declare variance yourself yet, but knowing what out means (producer, can be read but not written) prevents confusion when reading Kotlin stdlib and Jetpack signatures.
Coroutine internals — channels, select, coroutine context, custom coroutine scopes. You'll use viewModelScope and lifecycleScope without needing to understand how the coroutine machinery implements them. Learn this when you need to debug a cancellation issue or build a custom scope.
Skip entirely for now (~35%)
DSL builders · Reflection · Kotlin/JS specifics · expect/actual multiplatform declarations (Phase 6 problem) · Complex generic variance you'd write yourself · KSP annotation processor authoring · Metaprogramming · Kotlin scripting
Kotlin 2.x + K2 compiler — what changed
Android Studio Meerkat now supports Kotlin K2 mode for Live Edit and Compose Preview. KSP2 (annotation processing) is now stable. Practically: faster builds, clearer error messages, nothing you configure manually. Just use the latest Android Studio and it's already on. The language itself didn't break anything — K2 is a compiler rewrite under the hood.
Resources: Kotlin Koans for Tier 1 foundations. "Kotlin in Action" (2nd ed.) covers Tiers 1 and 2 with depth. The official Kotlin docs are good as a reference once you know what you're looking for — don't read them linearly.
Phase 1 Jetpack Compose + Material 3 Expressive ~2 weeks
Compose stableM3 Expressive — new 2025Live Edit
▼
@Composable functions — UI is a pure function of state. No XML, no findViewById. A screen is literally @Composable fun ProductScreen(state: ProductUiState).
State — remember + mutableStateOf — local UI state that survives recomposition but not navigation or rotation. Understand the difference between this and ViewModel state.
Core layouts — Column, Row, Box, LazyColumn, LazyVerticalGrid. These replace LinearLayout, ConstraintLayout, RecyclerView. Google is pushing adaptive layouts hard in 2025 — also learn LazyVerticalStaggeredGrid.
Modifier chain — sizing, padding, click handlers, background, drawing order. Learn the mental model, not every modifier by heart. The IDE autocomplete covers the rest.
Recomposition — Compose only re-renders composables whose observed state changed. Understanding this prevents subtle performance bugs. The new @FrequentlyChangingValue lint annotation (alpha, I/O 2025) now warns you proactively.
Material 3 components — Button, TextField, Card, TopAppBar, Scaffold, NavigationBar. Don't build from scratch what M3 gives you for free.
Material 3 Expressive — Google I/O 2025
Google released M3 Expressive, a major design update with new animated components, spring physics-based motion, richer haptic feedback, and deeper theming. It ships as an update to the existing M3 library — not breaking. Learn stable M3 first, then layer in Expressive components as they stabilise. This is what Android 16's system UI is built on, and it's the direction all new Compose UI is heading.

A product listing screen with a LazyVerticalGrid of product cards using hardcoded fake data, a top search bar, and a detail screen that opens on tap. No networking, no database — just composables rendering data. Looks like the browse screen of a real marketplace app.

Phase 2 Architecture — MVVM now, MVI next ~2 weeks
MVVMMVI gaining tractionUDFStateFlow
▼
MVVM core loop — ViewModel exposes StateFlow<UiState>. Composable collects it with collectAsStateWithLifecycle(). User actions call ViewModel functions. Zero business logic in composables — they just render and forward events.
Sealed UiState pattern — one sealed class per screen, exhaustively handled in a when block in the composable. The compiler forces you to handle every state — no runtime surprises.
One-time events with SharedFlow — navigation triggers, snackbar messages, dialogs. These are events, not state. SharedFlow(replay = 0) fires once and doesn't replay on recomposition.
viewModelScope — coroutines that live and die with the ViewModel. All your async work (network, database) launches here. This is the Android equivalent of your Spring Boot @Async or coroutine scopes on the backend.
MVI is the direction in 2025 — understand the difference
The 2025 consensus: MVVM for simple screens, MVI (Model-View-Intent) for complex interactive flows. In MVI, user actions are explicit Intent objects. The ViewModel reduces them into a single immutable State. No inconsistent multi-variable state bugs. Compose's reactive nature fits MVI naturally. For a checkout flow, cart state, or order management — MVI is the cleaner choice. For a settings screen — MVVM is enough. Many experienced developers use both in the same app. Start with MVVM, introduce MVI patterns where complexity demands it.

Phase 3 Navigation — Compose Nav 2 (stable) + watch Nav 3 ~1 week
Nav 2.8+ — typed routesNavigation 3 — alpha
▼
NavController + NavHost — the router. NavHost declares all routes. NavController executes navigation. One controller per app, accessed via LocalNavController or passed as a lambda.
Typed routes (Nav 2.8+) — define routes as @Serializable data classes, not raw strings. data class ProductDetail(val id: String). IDE-refactorable, type-safe, no string typos at runtime.
Argument passing rule — always pass IDs between screens, never full objects. The destination fetches what it needs. Avoids Parcelable boilerplate and keeps navigation lightweight.
Back stack + bottom navigation — popBackStack(), popUpTo(), launchSingleTop = true. Wire a NavigationBar to NavController. Tab state persists correctly when switching between tabs.
Navigation 3 — announced Google I/O 2025, not stable yet
Google announced Navigation 3 at I/O 2025. More Compose-idiomatic API, better adaptive layout support for foldables and tablets, and Compose Multiplatform target support. Still in alpha. Learn Navigation 2 now — it's stable and will handle all your Android work. Keep an eye on Nav 3 going stable (estimated late 2025 or early 2026) and plan to migrate when ready.
Milestone
A 3-tab app (Home, Search, Cart) with a product list → product detail flow using typed routes. Back button works correctly on every screen. Selecting a tab doesn't recreate the screen if it was already visited.
Resource: Google's Navigation Compose codelab. Use docs for Navigation 2.8+ — filter specifically for typed routes to avoid older string-based examples.
Phase 4 Data layer — Retrofit, Room, Hilt, DataStore ~3 weeks · the deepest phase
RetrofitRoom KMP-readyHiltDataStorePaging 3
▼
Repository pattern — single source of truth. ViewModel → Repository → Remote (Retrofit) or Local (Room). Never ViewModel → Retrofit directly. This pattern maps directly to a typical backend service layer.
Retrofit + kotlinx.serialization — define APIs as Kotlin interfaces with @GET, @POST annotations. Serialization parses JSON into data classes automatically. Conceptually identical to your Spring Boot REST clients.
Room — SQLite with full coroutine support. @Entity data classes, @Dao interfaces with suspend functions, @Database class. Room generates all SQL. Same conceptual model as JPA/Hibernate on a backend — entities, repositories, queries.
DataStore — replaces SharedPreferences. Fully coroutine-based, type-safe, no I/O on the main thread. Use this for auth token storage, user preferences, app settings.
Hilt (dependency injection) — @HiltViewModel, @Inject, @Module, @Provides, @Singleton. Learn the why before the annotations — it exists for testability and avoiding manual dependency wiring. This takes the longest to feel natural. Give it time and repetition.
Offline-first architecture — fetch from network → save to Room → serve UI from Room. Your app works without internet and refreshes when reconnected. Gold standard for any e-commerce or marketplace app.
Room + ViewModel + DataStore are now KMP-stable — I/O 2025
Google announced at I/O 2025 that Room, ViewModel, DataStore, and Paging now have stable Kotlin Multiplatform support. The Room code you write today can be shared with an iOS target later with minimal changes. An official "Migrating Room to Room KMP" codelab was released at I/O. Start with Android-only Room — the migration path is documented and straightforward.

Android 16 + 16KB page size — check before Play Store submission
Android 16 (releasing 2025) requires apps to support 16KB memory page size. Android Studio Meerkat now shows warnings for incompatible APKs automatically. Pure Kotlin apps using standard Jetpack libraries are generally already compatible. You'll only have issues with native C/C++ code or certain third-party SDKs. Check Razorpay's and Fast2SMS's SDK changelog for 16KB compatibility statements before your first Play Store submission.
This is where learning ends and shipping begins
Every concept from Phases 1–4 exists to make this phase faster. If you already have a working backend, you're not learning Android and building a server simultaneously — you're building a mobile client for something that already exists. That's a much easier problem than most Android beginners face.
Phase 6 — future horizon Kotlin Multiplatform — after your Android app ships Not now · but build Android with this in mind
KMP stableCompose MultiplatformiOS from Kotlin
▼
What KMP is — write business logic once in Kotlin, run it on Android and iOS. Not the UI (unless you use Compose Multiplatform) — just repositories, data models, network calls, Room databases. Google officially supports KMP since I/O 2024, production-stable since I/O 2025.
Why it matters — KMP gives you a path to native Android + iOS with shared Kotlin business logic — no full rewrite needed. Room, ViewModel, DataStore all KMP-stable as of 2025. Google Docs ships with KMP in production on iOS.
Compose Multiplatform (CMP) 1.10 — JetBrains' extension sharing Compose UI across Android, iOS, desktop, and web. iOS is now stable. Hot Reload is stable in CMP 1.10.0. If you know Compose for Android, you already know ~80% of CMP. Major companies including Shopify and Forbes ship it in production.
The play for your app — Ship Android first, validate the product, then use KMP to expand to iOS without a full rewrite. The clean architecture you build in Phases 1–5 (Repository pattern, ViewModel, separated layers) is exactly what KMP expects. No rearchitecting needed.
Don't learn KMP now — but you're already building for it
KMP is production-ready. But it's a layer of complexity on top of solid Android fundamentals. Ship the Android app first. Once it works and users validate the product, KMP is a low-risk addition — your Room and ViewModel code migrates with minimal changes. The architecture decisions you make today are already KMP-compatible.
